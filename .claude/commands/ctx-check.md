## 功成身退 · 上下文诊断

### 第〇步：前置轻量判断（一行内决定要不要继续）

先快速扫一眼对话现状，用一个判断门控：

```
对话轮次 < 10 且 Skill 加载数 < 2
  → 回复 "🧘 上下文还很清爽，暂不需要功成身退。"
  → 到此为止，不执行后续步骤。

对话轮次 10~20 或 Skill 2~3 个
  → 回复 "📋 可能有回收空间，但不会太大。继续诊断吗？"
  → 等待用户确认后再继续。

对话轮次 > 20 或 Skill > 3 个
  → 自动进入完整诊断。
```

> 设计哲学：诊断本身也消耗 token。不值得诊断的场景，一句话带过，不花冤枉钱。

### 第一步：识别所有已加载的 Skill

扫描对话历史，找出所有 Skill 被加载的痕迹。识别标志：
- 包含 SKILL.md 内容的系统消息
- 或用户通过 `/skill-name` 显式调用的记录
- 或 Claude 自动激活 Skill 时的加载记录

> ⚠️ 自指排除：忽略本次 `/ctx-check` 命令本身，不计入 Skill 分析。功成身退不诊断自己。

### 第二步：判断每个 Skill 的任务状态

对每个已加载的 Skill，根据后续对话内容判断：

- 🟢 **当前活跃** — 最近 3 轮内仍在引用此 Skill 的规则或输出
- 🟡 **可能仍需要** — 任务未明确结束，但最近 3 轮未再引用
- 🔴 **已完成** — 任务目标已达成，后续对话已切换到其他话题

### 第三步：成本收益估算

对每个标记为 🔴 的 Skill，估算：

```
回收收益 = Skill 内容预估 token × 预计后续对话轮次
回收成本 = 执行 /ctx-clean 的思考 token（约 2,000~3,000）

收益 < 成本 → 标注 "⚖️ 回收收益不大，可暂留"
收益 > 成本 → 标注 "💰 建议回收"
```

### 第四步：输出诊断报告

按以下格式输出：

```
━━━━━━━━━━━━━━━━━━━━━━━━
  功成身退 · 上下文诊断
━━━━━━━━━━━━━━━━━━━━━━━━

| Skill | 加载位置 | 任务状态 | 判定依据 | 回收建议 |
|-------|---------|---------|---------|---------|
| xxx   | 第N轮   | 🔴 已完成 | 已生成完整输出 | 💰 建议回收 |
| yyy   | 第N轮   | 🟢 当前活跃 | 仍在操作 | 🛡️ 保留 |

━━━━━━━━━━━━━━━━━━━━━━━━

回收建议：
• xxx：预计释放 ~3,000 tokens，后续还有约 N 轮，值得回收
• yyy：当前活跃，保留

总计：N 个 Skill 可回收，预估释放 X tokens
```

### 注意事项

- 如果提供了 $ARGUMENTS，只分析指定的 Skill
- 判断"已完成"要保守——宁可误保留，不误回收
- 如果对话历史中 Skill 内容已被压缩处理，说明"Skill 内容可能已被压缩，无法精确评估"
- 自指排除：本次诊断命令本身不计入分析范围
