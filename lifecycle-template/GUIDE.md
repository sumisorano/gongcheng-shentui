# 功成身退 · 生命周期模板使用指南

## 一句话说清楚

**给 Skill 加三个字段，功成身退就知道它什么时候该走了。**

---

## 设计哲学：Progressive Disclosure 的第四层

Anthropic 为 Skills 设计了一套 **Progressive Disclosure（渐进式披露）** 三层加载机制：

| 层 | 内容 | 加载时机 |
|---|------|---------|
| 第一层 | frontmatter（name + description） | 始终被扫描 |
| 第二层 | SKILL.md 正文 | 用户请求匹配 description 时加载 |
| 第三层 | references/scripts/assets | 正文明确引用时才加载 |

三层一层比一层重，一层比一层懒。这是 Skill **"怎么进来"** 的设计。

**功成身退是第四层——"怎么出去"：**

| 层 | 内容 | 回收时机 |
|---|------|---------|
| 第四层 | 已加载的 SKILL.md 正文 | 任务完成后，ctx-clean 回收全文，只留 retention_summary |

进来是渐进的（progressive disclosure），出去也是渐进的（progressive retirement）。**完整的 Skill 生命周期 = 四层加载 × 功成身退。**

另外，这篇文章还强调了一个关键观点：**SKILL.md 不是塞得越满越聪明，是越克制越聪明。** 2000 行的 mega-SKILL.md 每次触发都要全量加载，反而让模型注意力稀释。正确做法是拆成多个 300-500 行的独立 Skill。这个原则与功成身退互补——拆分让每个 Skill 更轻，功成身退让用完的 Skill 不赖着占坑。

---

## 两个生命周期类型

### `one-shot` — 一次性任务

**用完即走，完成可回收。**

适合：
- 代码审查（输出报告后结束）
- 单元测试生成（生成文件后结束）
- 部署检查（检查完输出结果后结束）
- 文档生成（生成文档后结束）
- 安全审计（输出审计报告后结束）

特点：任务有明确的终点，完成后后续对话通常切到别的主题。

**你在 SKILL.md 里填的 `completion_signal`，就是 ctx-check 判断"能不能回收"的依据。**

### `session` — 会话常驻

**一直留到会话结束。**

适合：
- 编码规范（每轮写代码都要遵守）
- 项目架构约定（整个会话都要引用）
- 团队术语表（随时需要查）
- 持续生效的约束（如"所有 API 返回格式统一"）

特点：没有明确终点，整个会话期间都可能被引用。

## 三个字段怎么填

### `completion_signal` — 怎么算完成

**核心原则：写具体，不要写模糊。**

| ❌ 太模糊 | ✅ 具体 |
|----------|--------|
| "任务完成" | "审查报告已输出，所有问题已列出" |
| "生成完毕" | "测试文件已写入磁盘，且用户表示接受" |
| "检查结束" | "部署检查清单全部 ✅，用户已确认" |

**为什么？** ctx-check 需要根据 `completion_signal` 在对话历史中匹配"这事做完了没有"。模糊的描述没法匹配。

### `retention_summary` — 回收后留什么

**核心原则：一句话说清"做了啥 + 结果咋样"。**

| ❌ 太简略 | ✅ 刚好 |
|----------|--------|
| "审查完成" | "已按 OWASP Top 10 完成代码审查，发现 2 个安全问题已在后续修复" |
| "已生成" | "已为 src/auth/login.ts 生成单元测试，覆盖 12 个用例" |
| "已完成" | "部署检查通过：测试 ✅、版本号已更新、无遗留 TODO" |

**为什么？** 回收后这条摘要就是 Skill 内容的替代品——模型要知道"这事做过了，结论是啥"，但不需记住完整 SKILL.md。

### `type` — 选哪个

**判断标准**：这个 Skill 的指令在后续对话中还会被引用吗？

- "会" → `session`
- "不会，任务做完就切话题" → `one-shot`

**不确定？** 选 `session`——宁可多占 token，不冒险丢失指令。

---

## 填完之后怎么验证

1. 把填好的 Skill 放到 `.claude/skills/<name>/SKILL.md`
2. 在 Claude Code 里调用它完成任务
3. 跑 `/ctx-check` —— 它应该能根据你的 `completion_signal` 识别出"已完成"
4. 跑 `/ctx-clean` —— 回收成功后应该只保留 `retention_summary`

## 常见误区

### 误区一：所有 Skill 都该用 one-shot

❌ 不对。编码规范、架构约定、持续约束这些应该用 `session`。

**原则：看 Skill 的内容是"流程"还是"规范"。** 流程有终点，规范没终点。

### 误区二：completion_signal 写得越详细越好

❌ 不是。`completion_signal` 是给 ctx-check 做判断用的，不是给用户看的。写一个**可验证的标志**就够了，不需要解释为什么。

### 误区三：retention_summary 就是 description 的复制

❌ 不一样。`description` 是"这个 Skill 干嘛的"（触发前），`retention_summary` 是"这个 Skill 干了什么"（回收后）。
